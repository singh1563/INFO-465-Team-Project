Screen Summaries:

* Home Screen: Provides navigation to course search, student registration, instructor schedules, and session enrollment, with a quick course lookup field and overview of typical user tasks.
* Course Search Screen: Allows users to search for courses by department, instructor, or course number and view course details such as credit hours, prerequisites, modality, and enrollment limits.
* Student Registration Screen: Allows students to log in, view enrolled sessions, and register for new sessions while showing validation examples for duplicate registrations and full courses.
* Instructor Schedule Screen: Allows instructors to log in and view their assigned course sessions, meeting times, modality, location, and enrollment counts.
* Session Enrollment Screen: Allows authorized users to view a selected course session, enrolled students, seat availability, and enrollment rules.

HTML/CSS/JS Elements or Techniques Used:

* HTML structure using header, nav, main, section, article, form, table, and footer elements.
* Shared CSS file used across all pages for consistent layout, colors, fonts, buttons, cards, navigation, and page spacing.
* Responsive layout using CSS grid, flexbox, and media queries.
* JavaScript used for the Home Screen quick course lookup form validation and redirect behavior.
* Relative hyperlinks used between all .html files so users can navigate through the full prototype in a browser.
* Button click alerts used to demonstrate login, search, registration validation, and roster loading behavior.

Design Challenges or Decisions:

* The team used consistent page names and navigation links so every screen could connect to the rest of the prototype.
* A shared CSS file was used to keep the look and feel consistent across the Home, Course Search, Student Registration, Instructor Schedule, and Session Enrollment screens.
* The design uses a simple card-based layout so students, instructors, and administrators can quickly understand where to go.
* Demo data was included to make each screen understandable without requiring a real database or backend system.
* Registration validation was represented through sample messages for full courses, duplicate registration, prerequisite checks, and advisor approval.
